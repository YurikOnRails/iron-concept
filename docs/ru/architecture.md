# Архитектура Titan Core

## Принципы {#принципы}

### I. Пути READ и WRITE священны

Баг в визуализации физически не должен иметь возможности отправить команду на машину.

```
READ path — только данные, только в одну сторону:
  Датчики → Rust Edge Agent → NATS → TimescaleDB → Phoenix → WebSocket → UI

WRITE path — явный, авторизованный, логируемый:
  Оператор → Auth + 2FA → Audit Log → Command Service → NATS → Edge Agent → ПЛК
```

В классических SCADA (WinCC, Wonderware, MasterSCADA) эти пути перемешаны в одном монолите. Это архитектурная ошибка которая делает систему одновременно небезопасной и хрупкой.

### II. Интеллект на границе сети

Edge агент рядом с ПЛК — не тупой ретранслятор. Он:

- **Фильтрует по deadband** — не публикует тег если значение не изменилось значимо
- **Буферизирует локально** — при обрыве сети сохраняет в SQLite, при восстановлении отправляет
- **Валидирует качество данных** — маркирует `GOOD / UNCERTAIN / BAD`
- **Работает автономно** — завод продолжает работать даже если сервер недоступен

```rust
// Deadband: публиковать только при значимом изменении
fn should_publish(last: f32, current: f32, deadband: f32) -> bool {
    (current - last).abs() > deadband
}
// Результат: 80–90% меньше сообщений в сеть
```

### III. Одна база данных

```
Традиционный подход:          Titan Core:
  PostgreSQL  (конфиги)    →    TimescaleDB
  InfluxDB    (временные)  →    (одна база)
  Redis       (текущие)    →    + ETS кэш в Elixir
  
3 точки отказа               1 точка отказа
3 синхронизации              0 синхронизаций
3 набора инструментов        1 набор инструментов
```

TimescaleDB — это PostgreSQL с расширением для временных рядов. Конфигурация и данные датчиков в одном запросе:

```sql
SELECT
    t.name,
    t.unit,
    last(v.value, v.timestamp) AS current_value,
    obj.name AS object_name
FROM tags t
JOIN objects obj ON t.object_id = obj.id
JOIN tag_values v  ON v.tag_id  = t.id
WHERE obj.name = 'Reactor 01'
GROUP BY t.id, obj.id;
```

### IV. Разработчик и инженер равноценны

Три уровня входа, одна система под капотом:

```
Уровень 1 — только мышь        (технолог, оператор)
Уровень 2 — YAML конфиг        (инженер КИПиА)
Уровень 3 — React + TypeScript (разработчик)
```

### V. Открытый исходный код — не опция

Проприетарная SCADA с лицензированием per-tag и обязательным сертифицированным партнёром — это не просто дорого. Это заложник. Titan Core — MIT лицензия. Форкни. Владей. Запускай вечно.

## Масштабирование до 100 000 тегов {#масштабирование}

Средний нефтехимический завод — 100 000 тегов, обновление каждую секунду. Наивный подход убивает систему при 5 000 тегах. Три уровня фильтрации делают это управляемым:

### Уровень 1: Deadband на Edge Agent

```
100 000 опросов/сек
    ↓ deadband фильтрация
~10 000 публикаций/сек
(−90% трафика до выхода из цеха)
```

80–90% тегов стабильны в любой момент времени. Температура которая стоит на 87°C и колеблется на ±0.1°C не генерирует ни одного сообщения при deadband = 0.5°C.

### Уровень 2: GenServer маршрутизация в Elixir

```
Каждый тег = один лёгкий Elixir процесс (~2KB RAM)
100 000 тегов = ~200MB RAM — это нормально

GenServer знает список подписчиков этого тега.
Обновление идёт только подписчикам — не broadcast всем.
```

### Уровень 3: Подписка клиента на видимые теги

```
Оператор открыл экран насосной станции
→ Phoenix Channel получает: ["pump_01.pressure", "pump_01.flow", ...]
→ 50–200 тегов на экране
→ Браузер получает обновления только этих тегов
→ Не 100 000 сообщений/сек — а 50–200
```

### Итоговые числа

```
NATS JetStream:        10–15 млн сообщений/сек
Phoenix WebSocket:     2 млн соединений на сервере
Latency тег→браузер:   5–15ms
TimescaleDB insert:    ~1 млн строк/сек с батчингом
Тренд за 30 дней:      3–8ms через continuous aggregates
```

## TimescaleDB vs PostgreSQL

Завод с 10 000 тегов, обновление раз в секунду:

```
10 000 × 86 400 сек × 365 дней = 315 миллиардов строк/год
```

| Операция | PostgreSQL | TimescaleDB |
|---|---|---|
| INSERT 10k тегов/сек | ~80k rows/sec, деградирует | ~1M rows/sec, стабильно |
| Тренд за 7 дней | 30–120 секунд | <1 секунды |
| Avg за год | минуты | миллисекунды |
| Место (год, 10k тегов) | 800GB–1.5TB | 80–150GB |
| Удаление старых данных | DELETE = блокировки | DROP chunk = мгновенно |
| Агрегаты | вручную | continuous aggregates |

TimescaleDB — это PostgreSQL расширение. 100% совместимый SQL. Те же драйверы, те же ORM, тот же `pg_dump`. Один новый концепт — `time_bucket()`. Всё остальное уже знакомо.

## Система тегов

Иерархия в NATS отражает физическую топологию завода:

```
plant                              → весь завод
  └── line_1                       → линия 1
        └── reactor_01             → реактор 1
              ├── temperature      → температура (float32, °C)
              ├── pressure         → давление (float32, bar)
              └── pump_running     → статус насоса (bool)

Полное имя: plant.line_1.reactor_01.temperature
```

YAML описание тега — единственное место где описана физика объекта:

```yaml
reactor_01:
  temperature:
    source: modbus://plc-01/holding/0x1000
    type: float32
    unit: "°C"
    scan_rate: 1s
    deadband: 0.5
    range: [0, 200]
    alarms:
      high:
        limit: 180
        priority: 2
        message: "Высокая температура реактора"
      high_high:
        limit: 195
        priority: 1
        message: "Критично — аварийный останов"
        action: emergency_shutdown
```

Команды (WRITE path) — отдельный файл, отдельный маршрут, отдельные права:

```yaml
# commands/reactor_01.yaml — физически отдельно от tags/
reactor_01:
  pump_start:
    target: modbus://plc-01/coil/0x0020
    type: bool
    requires_confirm: true
    requires_role: operator
    audit: true
```
