# Deployment Guide

IRON is designed to run on your hardware, in your network, under your control.
Cloud deployment is optional — not the default.

```
Mode 1 — Development    iron dev                    Docker Compose, local machine
Mode 2 — Local server   iron deploy --target local  Kamal 2, LAN server or PC
Mode 3 — Cloud (opt.)   iron deploy --target cloud  Kamal 2, Hetzner or any VPS
Mode 4 — Edge (OT)      iron deploy --target edge   Kamal 2, Raspberry Pi / mini-PC
```

Modes 2, 3, and 4 use identical tooling — [Kamal 2](https://kamal-deploy.org).
The only difference is the server IP and network location.
You can run all modes simultaneously: local server for the main application,
edge agent in the OT zone, cloud optionally for remote access.

---

## Why local-first

Industrial plants have legitimate reasons to keep operational data on-premise:

- **Data sovereignty** — sensor readings, alarms, and historian data stay inside the plant network
- **OT network isolation** — the server lives on the IT side (VLAN 20), reachable from the OT zone without internet
- **No internet dependency** — the plant keeps running if the internet goes down
- **Regulatory compliance** — some jurisdictions and industries prohibit sending operational data outside the facility

A $150 Beelink EQ12 or an existing office PC is a fully capable IRON server
for a plant with up to 50,000 tags. Cloud is not required.

## Mode 1 — Local Development

No server needed. Runs a complete IRON stack on your laptop or desktop
using Docker Compose. No PLC required — simulator runs by default.

```bash
iron new myplant
cd myplant
iron dev
```

Open `http://localhost:4000`. You have a working IRON instance with simulated data.

What `iron dev` starts:

```yaml
# docker-compose.yml (generated by iron new)
services:
  iron_web:
    image: ghcr.io/getiron/iron-web:latest
    ports: ["4000:4000"]
    environment:
      DATABASE_URL: postgres://iron:iron@db/iron_dev
      NATS_URL: nats://nats:4222
    depends_on: [db, nats, iron_core]

  iron_core:
    image: ghcr.io/getiron/iron-core:latest
    environment:
      NATS_URL: nats://nats:4222
      SIMULATE: "true"          # No PLC needed in dev mode

  db:
    image: timescale/timescaledb:latest-pg16
    environment:
      POSTGRES_PASSWORD: iron
    volumes:
      - iron_db:/var/lib/postgresql/data

  nats:
    image: nats:latest
    command: "-js"

volumes:
  iron_db:
```

```bash
# Add simulated sensors
iron simulate reactor_01 --temperature "sine:20:180:60s"
iron simulate pump_01 --status "cycle:on:30s:off:5s"

# Validate your tag config
iron validate

# Stop
iron dev --stop
```

When you are ready to deploy to a real server — nothing changes except the target.

## Mode 2 — Local Server (Primary Production Mode)

Deploy to any Linux machine on your local network: an office PC, a mini-PC
in the control room, or a dedicated server in the server room.

**Recommended hardware:**

| Option | Cost | RAM | Notes |
|---|---|---|---|
| Beelink EQ12 (Intel N100) | ~$150 | 16GB | Passive cooling — no fan failure risk |
| Any office PC or workstation | $0–300 | 8GB+ | Reuse existing hardware |
| Dedicated server (Supermicro, Dell R350) | $500–2,000 | 32GB+ | For large plants, 100k+ tags |

A Beelink EQ12 running IRON handles up to ~50,000 tags comfortably.
For a greenhouse with 40 sensors or a mid-size factory with 5,000 tags,
it is more than enough.

### Configuration

```yaml
# config/deploy.yml
service: iron
image: ghcr.io/getiron/iron

servers:
  web:
    - 192.168.1.100          # IP of your local server

proxy:
  ssl: false                 # No SSL needed on LAN
  # ssl: true                # Enable if you have a domain + cert
  host: iron.local           # Local hostname or mDNS name

registry:
  server: ghcr.io
  username: your-github-username
  password:
    - KAMAL_REGISTRY_PASSWORD

env:
  clear:
    PHX_HOST: iron.local
    NATS_URL: nats://nats:4222
  secret:
    - SECRET_KEY_BASE
    - DATABASE_URL
    - DATABASE_PASSWORD

accessories:
  db:
    image: timescale/timescaledb:latest-pg16
    host: 192.168.1.100
    port: "127.0.0.1:5432:5432"   # Not exposed outside the server
    env:
      secret:
        - POSTGRES_PASSWORD
    volumes:
      - /var/lib/iron/postgres:/var/lib/postgresql/data

  nats:
    image: nats:latest
    host: 192.168.1.100
    port: "127.0.0.1:4222:4222"   # Not exposed outside the server
    cmd: "-js --store_dir /data"
    volumes:
      - /var/lib/iron/nats:/data
```

### Secrets file

```bash
# .kamal/secrets  — never commit this file, add to .gitignore
KAMAL_REGISTRY_PASSWORD=ghp_your_github_token
SECRET_KEY_BASE=run_mix_phx_gen_secret_to_generate_this
DATABASE_URL=postgres://iron:yourpassword@127.0.0.1/iron_prod
DATABASE_PASSWORD=yourpassword
POSTGRES_PASSWORD=yourpassword
```

### Deploy

```bash
# First time — installs Docker, pulls images, starts everything
kamal setup
# or:
iron deploy --target local

# Takes 3–5 minutes on first run.
# Open http://192.168.1.100 or http://iron.local
```

```bash
# Subsequent deploys — zero downtime (< 60 seconds)
iron deploy --target local

# What happens:
# 1. Build new image
# 2. Push to registry
# 3. Run database migrations
# 4. Start new container (old keeps running)
# 5. Health check passes → old container stops
# If health check fails → old container keeps running, deploy aborted
```

```bash
# Operations
kamal app status            # Is it running?
kamal app logs              # Recent logs
kamal app logs --since 1h   # Last hour
kamal rollback              # Instant rollback to previous version
kamal app exec "bin/iron eval 'IRON.version()'"  # Run commands inside container
```

### No internet on the server?

If your server has no internet access (common in secure facilities):

```bash
# On a machine WITH internet access:
docker pull ghcr.io/getiron/iron-web:0.3.0
docker pull ghcr.io/getiron/iron-core:0.3.0
docker pull timescale/timescaledb:latest-pg16
docker pull nats:latest

docker save ghcr.io/getiron/iron-web:0.3.0 | gzip > iron-web-0.3.0.tar.gz
docker save ghcr.io/getiron/iron-core:0.3.0 | gzip > iron-core-0.3.0.tar.gz
# ... same for timescaledb and nats

# Transfer to server via USB drive or SCP over LAN
scp iron-web-0.3.0.tar.gz user@192.168.1.100:~/

# On the server:
docker load < iron-web-0.3.0.tar.gz
# Kamal detects image is already present and skips pull
```

### High availability (two servers)

For plants where downtime is not acceptable:

```yaml
# config/deploy.yml — two servers, automatic failover
servers:
  web:
    - 192.168.1.100    # Primary
    - 192.168.1.101    # Secondary

accessories:
  db:
    # Patroni handles automatic PostgreSQL failover between the two servers
    # Failover time: under 10 seconds
    # See: docs/ha-setup.md
    hosts:
      - 192.168.1.100
      - 192.168.1.101
```

Two Beelink EQ12 units: ~$300 total. Automatic failover.
This is the correct answer to "what if the server dies" for an industrial plant.

## Mode 3 — Cloud Deployment (Optional)

Use cloud deployment when:
- You need remote access to the dashboard from outside the plant network
- You are managing multiple sites from a central location
- You are a system integrator providing IRON as a managed service to clients
- The plant has no on-premise IT infrastructure

The configuration is identical to Mode 2 — only the server IP changes.

**Recommended: Hetzner dedicated server**

| Server | Specs | Cost |
|---|---|---|
| Hetzner AX41 | AMD Ryzen 3600, 64GB RAM, 2× 512GB NVMe | €38/month |
| Two AX41 (HA) | Same × 2 | €76/month |

```yaml
# config/deploy.yml — cloud variant
servers:
  web:
    - 203.0.113.10          # Hetzner server IP

proxy:
  ssl: true                 # Let's Encrypt, automatic
  host: myplant.example.com # Your domain
```

```bash
# First deploy
kamal setup

# Dashboard at https://myplant.example.com
# SSL certificate obtained automatically from Let's Encrypt
```

Everything else is identical to Mode 2.

**Local + cloud simultaneously:**

```yaml
# config/deploy.yml — local primary, cloud for remote access
servers:
  web:
    hosts:
      - 192.168.1.100       # Local server — primary, always running
    proxy:
      host: iron.local
      ssl: false

  remote:
    hosts:
      - 203.0.113.10        # Cloud — remote access only
    proxy:
      host: myplant.example.com
      ssl: true
```

Both connect to the same NATS and TimescaleDB (on the local server).
The cloud instance is a read-only view — it does not store data.

## Mode 4 — Edge Agent (OT Network)

The edge agent (`iron-core`) runs physically inside the OT zone — in the control
cabinet or directly on a DIN-rail computer next to the PLCs.

It polls PLCs on the OT side and publishes to NATS on the IT side (your local server).

```
OT Zone (VLAN 10):  PLC ←→ Edge Agent (Raspberry Pi)
                                  │
                          NATS publish (one direction)
                                  │
IT Zone (VLAN 20):         NATS → TimescaleDB → iron-web → Browser
```

### Edge hardware

| Device | Cost | Notes |
|---|---|---|
| Raspberry Pi CM4 (industrial carrier) | ~$150 | -40°C to +85°C, ARM64 |
| Beelink EQ12 | ~$150 | Passive cooling, x86_64 |
| CLICK PLUS PLC | ~$200 | Linux inside — agent runs on same device |

### Configuration

```yaml
# config/deploy.yml — add edge section to your existing config
servers:
  web:
    - 192.168.1.100         # Local server (IT zone)

  edge:
    hosts:
      - 192.168.10.5        # Edge device IP (OT/IT bridge)
    cmd: /app/bin/iron_core
    options:
      network: host         # Direct access to OT network interfaces
    env:
      clear:
        NATS_URL: nats://192.168.1.100:4222   # Your local server
        TAGS_CONFIG: /app/config/tags.yaml
        BUFFER_PATH: /var/iron/buffer.db      # SQLite: survives NATS loss
```

```bash
# Deploy edge agent to Raspberry Pi
iron deploy --target edge-01
# or:
kamal deploy --hosts 192.168.10.5

# Kamal handles ARM64 automatically via multi-arch Docker image
# Restart takes ~4 seconds
# SQLite buffer preserves all data collected during restart
```

### Air-gapped OT networks

Many OT networks have no internet access. The edge device cannot pull from Docker Hub.

```bash
# On a machine with internet access:
docker pull ghcr.io/getiron/iron-core:0.3.0 --platform linux/arm64
docker save ghcr.io/getiron/iron-core:0.3.0 | gzip > iron-core-arm64-0.3.0.tar.gz

# Transfer via USB or from IT zone (SCP over the VLAN boundary):
scp iron-core-arm64-0.3.0.tar.gz user@192.168.10.5:~/

# On the edge device:
docker load < iron-core-arm64-0.3.0.tar.gz
# Kamal skips pull — image already present
```

## Upgrade Procedure

```bash
# Check current version across all targets
iron version --target local
iron version --target edge-01

# Upgrade local server (zero downtime)
iron deploy --target local
# Old container runs until new one passes health check
# If migration or health check fails: automatic rollback

# Upgrade edge agent
iron deploy --target edge-01
# ~4 second restart
# Buffer preserves data during restart — no data loss
```

Database migrations run automatically before the new container starts.
Migrations must be backwards-compatible with the running version.
`iron` enforces this — it will refuse to deploy a migration
that is not flagged as backwards-compatible.

## Environment Variables Reference

### iron-web

| Variable | Required | Description |
|---|---|---|
| `SECRET_KEY_BASE` | ✅ | Phoenix secret key (run `mix phx.gen.secret`) |
| `DATABASE_URL` | ✅ | PostgreSQL connection string |
| `NATS_URL` | ✅ | NATS server (`nats://host:4222`) |
| `PHX_HOST` | ✅ | Hostname for cookie security |
| `IRON_TELEGRAM_TOKEN` | No | Bot token for alarm notifications |
| `IRON_ADMIN_EMAIL` | No | Creates initial admin account on first boot |

### iron-core (edge agent)

| Variable | Required | Description |
|---|---|---|
| `NATS_URL` | ✅ | NATS server to publish to |
| `TAGS_CONFIG` | ✅ | Path to `tags.yaml` |
| `BUFFER_PATH` | No | SQLite buffer path (default: `/var/iron/buffer.db`) |
| `LOG_LEVEL` | No | `debug` / `info` / `warn` / `error` (default: `info`) |

## Troubleshooting

```bash
# Container not starting
kamal app logs --since 10m

# Database issues
kamal accessory logs db --since 30m

# NATS not reachable from edge agent
# Check: firewall rule OT→IT zone on port 4222?
kamal app exec --host 192.168.10.5 "bin/iron_core nats ping"

# SSL certificate not issuing
kamal proxy logs
# Check: does your domain DNS point to the server?
# Let's Encrypt requires port 80 open for HTTP challenge

# Edge agent not receiving PLC data
kamal app exec --host 192.168.10.5 "bin/iron_core modbus ping 192.168.10.100"
# Check: is the PLC IP reachable from the edge device?
# Check: is Modbus port 502 open on the PLC?
```

## Deployment Model Comparison

| | Mode 1 Dev | Mode 2 Local | Mode 3 Cloud | Mode 4 Edge |
|---|---|---|---|---|
| **What runs where** | Your laptop | Plant server | Hetzner VPS | OT cabinet |
| **Data location** | Local only | Plant network | Cloud provider | OT network |
| **Internet needed** | No | No | Yes | No |
| **SSL** | No | Optional | Automatic | No |
| **Cost** | $0 | $0–2,000 (hw) | €38+/month | $150 (hw) |
| **Best for** | Development | Production default | Remote access / multi-site | PLC polling |
| **OT network access** | No | Possible | No | Yes (native) |
